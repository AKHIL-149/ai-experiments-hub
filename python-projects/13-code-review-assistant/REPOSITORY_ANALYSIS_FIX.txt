===========================================
ROOT CAUSE ANALYSIS
===========================================

1. INITIAL BUG (Commit 13.5.28):
   - CodeAnalyzerService was incorrectly initialized with a session parameter
   - Result parsing was looking for result['issues'] instead of result['report']['issues']
   - Celery worker wasn't listening to 'analysis' queue

2. DATABASE SCHEMA BUG (Commit 13.5.29):
   - CodeFile table required pull_request_id (NOT NULL)
   - Repository analysis doesn't have a PR
   - Issues couldn't be saved to database

TECHNICAL DETAILS
===========================================

Schema Changes:
```sql
-- Made pull_request_id nullable
ALTER TABLE code_files MODIFY COLUMN pull_request_id VARCHAR(36) NULL;

-- Added repository_id for repository-level analysis
ALTER TABLE code_files ADD COLUMN repository_id VARCHAR(36);

-- Created index for repository queries
CREATE INDEX idx_file_repo_language ON code_files (repository_id, language);
```

Worker Flow:
1. Task triggered: analyze_repository_task(repository_id)
2. Discover files: Find all .py, .js, .ts, .java, .go, .rs files
3. Exclude: node_modules, venv, dist, build, __pycache__, .git
4. For each file:
   a. Read file content
   b. Analyze with CodeAnalyzerService
   c. Create CodeFile record with repository_id
   d. Create Issue records linked to CodeFile
   e. Commit to database
5. Return summary: files_analyzed, issues_found
